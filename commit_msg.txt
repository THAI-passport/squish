feat: in-browser PDF previews, visual page picker, editable merge name

Add client-side PDF rendering (vendored pdf.js) and richer file interactions
across the UI, plus a smarter merge output name. No document ever leaves the
browser — pdf.js renders locally and is served from the app, not a CDN.

UI (backend/static/index.html)
- Row thumbnails: multi-file tools show a first-page preview per file, lazily
  rendered, falling back to an icon if pdf.js is absent or a file won't parse.
- Drag-to-reorder file rows for multi-file tools; keyboard move button kept as
  the accessible path.
- Visual page picker for single-PDF page tools (split, remove-pages, reorder,
  rotate, crop, render, …): a grid of every page with three registry-driven
  modes — select (keep), remove (red), and order (drag to arrange). The grid
  and the page-range box sync both ways; the box stays authoritative on submit
  and is the keyboard path. Lazy render via IntersectionObserver, with an
  opt-in prompt above 600 pages. Picker tears down (destroys the pdf.js doc,
  removes listeners) on tool/file change.
- Editable merge output name, pre-filled from the input stems (report.pdf +
  invoice.pdf -> report+invoice.pdf), overridable.

Backend (tools.py)
- merge honors output_name; output_pdf_name() sanitizes via safe_component and
  guarantees the .pdf extension; merge_default_name() mirrors the UI default.
- pages fields carry a `picker` attr (select/remove/order) serialized through
  /api/tools so the picker is registry-driven — no per-tool frontend code.

Vendoring & deployment
- pdf.js vendored under backend/static/vendor (fetch-pdfjs.sh, README); fetched
  at Docker build and by run-local.sh; gitignored blobs; icon fallback if absent.
- Fix: load pdf.js from /vendor/ not /static/vendor/. StaticFiles is mounted at
  root, so /static/... 404'd and silently disabled all previews.

Tests

  custom + fallback).
